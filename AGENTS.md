# FO Test Harness — Agent Instructions

This repo contains the FounderOps BUILD → QA → DEPLOY harness plus intake tooling.
Use this file as the base operating guide for any agent working in this codebase.

## What This Repo Does
- Converts founder answers into structured intake JSON.
- Runs a Claude BUILD + ChatGPT QA loop to produce artifacts.
- Packages results into a ZIP under `fo_harness_runs/`.
- Supports full BUILD → QA → DEPLOY pipeline with auto-resume behavior.

## Pipeline Overview (Three Stages)
| Stage | Script | Purpose |
|-------|--------|---------|
| 1. Intake | `intake/generate_intake.sh` | Founder answers → structured intake JSON |
| 2. Build-QA | `run_integration_and_feature_build.sh` | Phase-by-phase BUILD + QA + integration check |
| 3. Deploy | `deploy/zip_to_repo.py` → `deploy/pipeline_deploy.py` | ZIP → GitHub → Railway / Vercel |

## Pre-Intake Gap Analysis (Pass0)
Use `gap-analysis/` to convert a raw idea into a build-ready brief, pricing, and hero JSON.

```
./gap-analysis/run_full_pipeline.sh /path/to/preintake.json --verbose
```

## Key Entry Points
- `fo_test_harness.py`: Main orchestration pipeline (BUILD → QA → optional DEPLOY).
- `intake/run_intake_v7.sh`: Generates intake JSON from hero answers.
- `intake/convert_hero_answers.py`: Converts raw founder text into hero JSON.
- `check_boilerplate_fit.py`: Analyzes whether a startup fits the boilerplate.
- `run_integration_and_feature_build.sh`: Main greenfield pipeline with phase builds + integration checks.
- `add_feature.sh`: Adds a single feature to an existing ZIP or repo.

## Common Workflows
1. Intake generation:
   - `cd intake`
   - `./generate_intake.sh hero_text/<startup>.json`

2. Build + QA loop:
   - `python fo_test_harness.py --intake <intake.json> --startup-id <id> --block B --build-gov <zip> --tech-stack <custom|lowcode>`

3. Review outputs:
   - `fo_harness_runs/<startup>_BLOCK_<X>_<timestamp>/`

4. Greenfield full pipeline (recommended):
   - `./run_integration_and_feature_build.sh --intake intake/intake_runs/<startup>/<startup>.json --startup-id <startup>`

5. Add a feature to an existing build:
   - `./add_feature.sh --intake <intake.json> --feature "<feature name>" --existing-zip <zip>`
   - or `--existing-repo <path-or-github-url>`

6. Pre-intake gap analysis:
   - `./gap-analysis/run_full_pipeline.sh /path/to/preintake.json --verbose`

## Safety & Repo Hygiene
- Never commit or expose API keys. Keys should only be in environment variables.
- Treat `.claude/settings.local.json` as sensitive; do not copy keys from it.
- Avoid modifying `fo_harness_runs/` and `boilerplate_checks/` unless explicitly asked.
- Preserve build artifacts and QA logs when changing the harness.
- Directive: Do not write or modify code unless you receive an explicit “YES” from the user.
- Always commit `fo_run_log.csv` when it changes.
- Outside of actual builds, default to ChatGPT instead of Claude to control costs.
- Safety prompt required: Before modifying `fo_run_log.csv` or `fo_test_harness.py`, explicitly confirm with the user.

## QA Gates (Summary)
Five sequential gates run inside `fo_test_harness.py`, then a post-build integration check:
1. COMPILE (deterministic AST parse)
2. STATIC (deterministic structural checks)
3. CONSISTENCY (AI, pre-filter only)
4. QUALITY (AI)
5. FEATURE_QA (AI)
6. INTEGRATION (deterministic `integration_check.py`, post-build)

**Rule:** CONSISTENCY is a pre-filter, not the authority. After 4 consecutive CONSISTENCY failures, fall through to QA; do not trigger a full rebuild based solely on CONSISTENCY.

## Integration Check (Post-Build)
`integration_check.py` runs 15 deterministic checks across backend and frontend and is used automatically after feature builds. Its output can be fed back into `fo_test_harness.py` via `--integration-issues <file>` for targeted fixes.

## Cleanup
Use `cleanup_fo_harness_runs.py` to reduce disk usage under `fo_harness_runs/` while keeping `.zip` files and recent runs. It can remove heavy subfolders (`build/`, `qa/`, `logs/`, `tmp/`) from older runs.
